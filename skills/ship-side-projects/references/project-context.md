# Project Context Snapshot

Create a concise snapshot only when authoritative repository guidance is missing, fragmented, or repeatedly misapplied. Prefer repairing or linking the established source of truth. Never let a generated snapshot override repository instructions or observed reality.

Capture:

- product goal, primary users, and current outcome signal;
- authoritative sources and unresolved contradictions;
- technology stack and supported versions;
- architecture/domain boundaries and critical decisions;
- security, privacy, data, accessibility, and compliance constraints;
- exact build, test, lint, type-check, and local-run commands;
- deployment, observability, migration, compatibility, and recovery constraints;
- durable user-approved preferences relevant to implementation.

Label observations, documented rules, decisions, and assumptions separately. Link rather than duplicate detail. Review the snapshot when architecture, tooling, policy, or repeated agent behavior shows it is stale.

Use `assets/templates/project-context.md` when a durable snapshot is justified.
