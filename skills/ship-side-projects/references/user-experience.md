# Continuous User Experience Collaboration

Activate the UX Designer perspective for every feature that changes what a person sees, hears, reads, enters, chooses, understands, or recovers from. This includes graphical interfaces, command-line interaction, notifications, onboarding, settings, errors, destructive actions, and human-facing API or automation behavior. Scale the depth to the change; do not reduce UX to visual styling.

## Collaborate through the increment

1. **Frame together:** the Requirements Engineer and UX Designer identify the user, context, job or journey, current friction, evidence, constraints, and success signal. They create the initial evidence-tagged persona and matching User Agent context, then Requirements Engineering hands maintenance accountability to UX and exits. Separate the underlying need from the proposed interface.
2. **Define behavior together:** agree information hierarchy, terminology, primary and recovery paths, and relevant loading, empty, partial, error, success, permission, offline, destructive, and interruption states. Translate them into testable usability and accessibility criteria.
3. **Reduce uncertainty:** use the cheapest artifact that can change a decision—annotated flow, content sketch, wireframe, interactive prototype, existing-pattern comparison, or direct observation. Skip mock-ups only when the established pattern and criteria already make the interaction clear; record that reasoning briefly.
4. **Build together:** have development surface feasibility, platform, performance, data, security, and maintenance constraints before they become accidental product decisions. Let UX review the running implementation at meaningful checkpoints and update criteria when evidence changes the design.
5. **Verify together:** exercise the complete implemented journey, including keyboard and assistive behavior where relevant, responsive constraints, content clarity, failure/recovery, and harmful edge cases. After applicable gates pass, give a fresh-context User Agent the current persona, bounded missions, criteria, exact revision, and access to the real app. It reports reproducible observations independently to Product Stewardship and UX before builder acceptance.
6. **Learn continuously:** UX records and dispositions every relevant learning from the User Agent, research, observed use, support, analytics, and builder decisions, updating the persona and next User Agent context when warranted and retaining source, date, confidence, and contradictions. User Agent output may expose assumptions but cannot by itself prove persona attributes. When the feature serves users beyond the builder, seek proportionate real-user observation or feedback and feed findings into the next bets. Do not invent target-user reactions.

The builder retains product and material design authority. The UX Designer may reject UX readiness when the journey or criteria are materially unresolved, and the Software Developer may reject implementation feasibility, but neither silently decides the other's accountability. Escalate material tradeoffs with options, consequences, evidence, and a recommendation.

The User Agent is a challenger, not a proxy customer. Keep its prompt free of implementation rationale and desired conclusions. Product Stewardship dispositions value and scope findings; UX dispositions usability/design findings and owns persona/context changes. Record disagreements rather than letting either role silently discard feedback.

For a trivial copy correction or established-pattern adjustment, a concise lead-held UX pass may be enough. For a new or consequential workflow, accessibility-sensitive interaction, unfamiliar user group, destructive action, or evidence-poor design, use a focused UX specialist and stronger prototyping or observation when available and proportionate.
